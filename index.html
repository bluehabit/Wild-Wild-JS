Number.EPSILON

Special Note Numbers that are not integers are often referred to as floating point numbers. Operations on these numbers may not always result in the exact value because computers have difficulties representing them. For example, 0.6 + 0.3 evaluates to 0.8999999999999999 instead of 0.9. One way to fix such logic is to calculate the difference between the actual result and the expected result and determine whether the difference is very small, i.e., close to zero. In JavaScript, Number.EPSILON represents a number that is very close to zero. We can check if the difference is smaller than Number.EPSILON to verify that it is small enough. 

		
var x = 0.6;
var y = 0.3;
var diff = (x + y) - 0.9; //the difference between (x+y) and 0.9
var absDiff = Math.abs(diff); // absolute difference
//the absolute difference is very small, meaning (x+y) and 0.9 are almost equal

if (absDiff < Number.EPSILON){ 
		console.log("Inside if");
}

In this example, the variable diff holds the difference between x + y and 0.9. Then we calculate the absolute difference using the Math.abs(...) function. If this absolute difference is smaller than Number.EPSILON, we can say that x + y and 0.9 are almost equal.

Callback Functions AKA Higher Order Functions

Callback functions are part of the functional programming paradigm, and callback functions themselves are one of the largest corner stones of functional programming

Callbacks are very important because it allows us to use closures to access variables that would have normally been destroyed. It is very important to understand in javascript functions are just values you can pass around.

Remember local variables inside of a function, only exist within that functions scope. They are created and destroyed when that function executes. Callbacks let us preserve these variables to be later used inside other functions!

Anytime you see an inline anonymous function, thats a clue right away that should alert you that its a callback function

Note, callback functions are not executed immediately, they are 'called back'(hence the name) at a specified point inside the higher order functions body.

Callback functions are closures

When we pass a callback function as an argument to another function, the callback is exectued at some point inside the containing functions body as if the callback were defined in the containing function. This means callbacks are closures. Closures have access to the containing functions scope, so the callback function can access the containing functions variables, and even variables from the global scope

Every tag you see on a page is a node, there are also different types of nodes: Element Nodes, Text Nodes, and Attribute Nodes

But what happens here?

		
	var dog = {
		sound: 'woof',
		talk: function(){
			console.log(this.sound);
		}
	}

	dog.talk() //woof

	var talkFunction = dog.talk
	talkFunction() // undefined

Whaaaatt? Why is this returning undefined? Because when you assign a method to a variable, in this case dog.talk, it is no longe a method that belongs to an object. Now it is just a regular function.

It is no longer a method connected to an object - its just a free floating function. It has lost its connection. You could fix this by doing dog.sound instead of this.sound in the original object. But thats not very useful lets use bind instead

To put it another way, lets rewrite the function to look like this

		
		var talkFunction = function(){
			console.log(this.sound) 
			/* there is no this in this context. There is no object invoking the function, that is what this refers to it refers to the context that it is being called */
		}

		talkFunction(); //undefined

This always is in reference to the object that invokes it. So the value of this depends on the context where the function is being CALLED. However we can BIND it to that context by using the function bind as shown below. When we use bind we make a copy of the function, and bind it to the proper context. Where the function was CALLED gives us its CONTEXT

		
		var dog = {
		sound: 'woof',
		talk: function(){
			console.log(this.sound);
			}
		}

		dog.talk() //woof

		var talkFunction = dog.talk;
		var boundFunction = talkFunction.bind(dog);
		//so bind forces 'this' keyword to be dog in this case
		boundFunction(); //woof

Think about how we use the word this in the english language. The word this by itself doesn't mean anything without context

Imagine you turn to your friend and say I don't like this, they won't know what your talking about, there is no context. Bind is a way we can be very specficic in what we mean by 'this'

Variable Hoisting

		
	var myVar = 'my Value';

	(function(){
		alert(myVar);
		//undefined

		var myVar = 'local value';
		//this is a local variable

		alert(myVar)
		//'local value'
	})();

Why is the output different for myVar in the above example? Its because variables get hoisted to the top of their scope. Don't let the word hoist confuse you. All it means is behind the scenes if you have a variable declared in a function scope its going to be brought to the very top, BUT just the declaration not the initialization.

Here is whats happening at the top

		
		var myVar = 'my Value';

		(function(){
			var myVar; //behind the scenes declaration
			alert(myVar);
			//undefined

			var myVar = 'local value'; //it would initialize here
			//this is a local variable

			alert(myVar)
			//'local value'
		})();

This is why if you read a book like JS the Good parts it always recommends you declare your variables at the top of a function, this is why.

So to reiterate if you have variable declarations scattered throughout your function, get rid of those, and at least declare them at the top of your function
